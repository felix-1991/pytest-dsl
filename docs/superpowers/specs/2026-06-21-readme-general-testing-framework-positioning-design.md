# README 通用自动化测试框架定位设计

## 目标

将 README 从“以 API 接口测试为主要产品定义”重构为“通用关键字驱动自动化测试框架”。API 测试保留为最容易上手、内置能力最完整的首个实践入口；DSL 表达、关键字扩展和 pytest 生态共同定义框架边界，使复杂业务流程、集成测试和端到端测试成为主叙事的一部分，而不是后置补充。

## 核心定位

首页使用以下三层定位：

1. pytest-dsl 基于 pytest，用统一 DSL 表达测试步骤、参数、变量、控制流和关键字调用；
2. 内置 HTTP 关键字提供低成本、可直接运行的 API 测试入口；
3. DSL、Python、插件和 XML-RPC 关键字可封装外部驱动，组合复杂测试和端到端业务流程。

标题采用“用关键字组织从接口到端到端的自动化测试”，避免把 API 测试写成框架的能力上限。

## 信息结构

README 按以下顺序组织：

1. **首屏定位**：先定义通用关键字驱动框架，再说明 API 是开箱即用的切入点，并给出端到端扩展方向；
2. **为什么是 pytest-dsl**：说明 DSL 统一结构、业务可读步骤、关键字复用和 pytest 生态价值，不再用一大段 API Python 代码定义框架价值；
3. **框架如何工作**：解释“DSL 业务流程 → 内置或自定义关键字 → Python 库和外部驱动 → pytest 执行与 Allure 报告”的主链路；
4. **从 API 开始**：保留可运行的 `[HTTP请求]` 示例和 mock 服务，把它明确标注为最快上手路径；
5. **扩展到复杂测试与端到端**：说明 DSL、`.resource`、项目 `keywords/`、插件和 XML-RPC 的组合方式，并给出标注为示意的跨层流程；
6. **执行结果与工具**：说明可读报告和 Studio 如何服务于所有 DSL 测试，现有 API 截图只作为具体演示；
7. **快速开始**：保留 API quickstart，因为它是当前最完整、可离线验证的入门项目；
8. **核心能力**：按 DSL 测试表达、关键字扩展与组合、HTTP/API、执行报告与工具排序。

## 示例策略

- 保留仓库中已经验证的 API quickstart，避免首页失去可复制、可运行的完整示例；
- 将当前冗长的“常规 pytest API 代码对比”替换为更短的框架执行模型说明；
- 端到端流程只使用“团队自定义页面操作关键字”“团队自定义数据校验关键字”等明确标签，不伪装成内置能力；
- 不新增无法在仓库中运行的完整 UI 或移动端示例，不使用不存在的内置关键字名称制造能力假象。

## 能力边界

- pytest-dsl 提供测试表达、关键字组织、执行和报告基础设施；
- HTTP/API 是内置领域能力，不代表框架只适用于接口测试；
- 浏览器、移动端、数据库、消息队列等技术动作需要团队通过 Python、插件或远程关键字接入对应驱动；
- 与 Robot Framework 的共同点只限于关键字驱动的组织和扩展思路；pytest-dsl 不兼容其语法、库接口或现有用例，也不宣称是直接替代品；
- 保留 pytest 原生收集、插件和命令行生态的准确表述，不扩大未验证的 fixture 集成能力。

## 文案原则

- 使用“低成本上手”“开箱即用”代替“零门槛”；
- 先说明框架能组织什么，再说明内置 HTTP 能力如何降低首个用例成本；
- 用事实描述扩展机制，避免“完美兼容”“发挥到极致”等不可验证表述；
- README 保持产品首页密度，详细参数和教程继续链接到专题文档；
- 控制全文长度，避免因新增通用定位而保留所有旧 API 解释造成重复。
